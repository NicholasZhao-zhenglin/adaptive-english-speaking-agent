# 工程学习记录

## 项目目标与背景

将个人 AI 助手中的自适应英语表达学习模块抽离为独立项目，保留原项目代码不动。新项目需要具备独立运行、测试、配置、文档和发布能力，并与点子、日记、待办等其他助手完全解耦。

## 当前架构与关键模块

- `english_learning_agent.py`：学习画像、表达级掌握度、错因、间隔复习、每日规划和诊断。
- `english_assistant.py`：调用 LLM 生成表达与练习，执行历史去重和缓存。
- `app.py`：只暴露英语 Agent API，并提供同源前端。
- `frontend/`：展示计划、内容、四类练习和反馈同步。
- `generate_daily.py`：供本地任务计划或手动运行的每日生成入口。

## 决策记录

### 2026-07-23｜步骤：确定独立仓库边界

- 目标：抽取可独立开发的英语 Agent，同时不复制其他助手或个人运行数据。
- 做了什么：只迁移英语决策层、内容生成工具和相关测试；重新实现仅含英语路由的 Flask 应用与独立前端。
- 观察到的事实：原 `app.py` 同时导入八类助手，前端脚本和样式也包含全部模块；直接复制会保留大量无关耦合。
- 为什么这样做：独立项目的价值是边界清晰，后续语音、评测和存储演进不会被个人助手其他功能牵制。
- 考虑过的替代方案及取舍：保留完整前后端再隐藏其他模块速度更快，但会造成死代码和错误的依赖边界；本次选择重新收口应用层。
- 遇到的问题/失败尝试：最初尝试在 `D:\` 创建兄弟目录，被当前工作区权限拒绝；改为在原工作区内建立嵌套但拥有独立 `.git` 的仓库，并在父仓库本地 exclude。
- 如何验证：新增边界测试，确认健康检查返回独立服务名，点子、待办和日记路由均为 404。
- 结论：新仓库运行边界只包含英语 Agent，原仓库文件未删除或修改。
- 我应该学会：代码“复制出来”不等于完成模块化；真正的拆分要从入口、配置、数据和测试四个边界同时解除耦合。

### 2026-07-23｜步骤：建立测试优先的独立后端

- 目标：让迁移后的决策层在新的 Flask 入口下保持行为一致。
- 做了什么：先运行缺少独立 `app.py` 的测试并得到失败，再实现英语 API、静态页面服务、请求大小限制和安全响应头。
- 观察到的事实：原项目依赖本机 `D:\Lib\site-packages` 路径，新仓库使用标准 `requirements.txt` 和 `.env`，不再依赖特定机器路径。
- 为什么这样做：可复现依赖和环境配置是独立仓库可维护性的基础。
- 考虑过的替代方案及取舍：继续使用全局 Python 环境部署更省步骤，但他人无法可靠复现；选择虚拟环境和显式依赖。
- 遇到的问题/失败尝试：无。
- 如何验证：15 条 Python 单元与接口测试通过，包括画像、调度、掌握度、生成上限、API 契约、异常 JSON 契约和项目边界。
- 结论：核心行为在新入口下保持稳定，且不需要导入原项目其他助手。
- 我应该学会：拆分时先把“必须继续成立的行为”写成测试，迁移才不会退化为手工复制后的主观确认。

### 2026-07-23｜步骤：重建最小独立前端

- 目标：保留 Agent 的完整学习闭环，同时消除对原个人助手 DOM 和路由的依赖。
- 做了什么：新建专用页面，支持画像编辑、计划解释、历史切换、内容生成、选择/填空/场景/配对练习和表达级结果回传。
- 观察到的事实：原前端脚本超过两千行并在启动时绑定多个助手 DOM，无法在删除其他页面后直接运行。
- 为什么这样做：专用前端比条件判断和空 DOM 兼容层更容易维护，也更能体现 Agent 项目的主体。
- 考虑过的替代方案及取舍：复制全部 CSS/JS 改动更少，但会把无关功能和死样式带入新仓库；选择围绕现有 API 重新实现专用界面。
- 遇到的问题/失败尝试：无。
- 如何验证：前端边界测试确认不包含其他助手名称或 API；JSON 响应回归测试和 JavaScript 语法检查通过。
- 结论：前端只依赖 `/api/english/*`，可以随独立仓库继续演进。
- 我应该学会：前端模块化不仅是拆文件，还要避免全局初始化对无关 DOM 的隐式依赖。

## 问题、失败尝试与解决方案

- 新仓库外层目录受工作区权限限制：改为工作区内嵌套独立 Git 仓库，并通过父仓库本地 exclude 隔离。
- 原项目依赖硬编码的本机 Python 路径：新项目删除该路径，使用虚拟环境和 `requirements.txt`。
- 原前端包含大量其他助手逻辑：以英语 API 契约为边界重新实现专用前端。

## 验证与结果

- Python：15 条单元与接口测试通过。
- Node.js：API 响应与前端边界测试通过。
- JavaScript：`api_response.js`、`script.js` 语法检查通过。
- 安全边界：未复制 `.env`、运行日志、个人学习 JSON 或任何 API Key。
- 真实运行：健康、仪表盘和历史接口均为 200；浏览器桌面与移动端通过，控制台无 error/warn。

### 2026-07-23｜步骤：完成可复现环境与浏览器验收

- 目标：证明新仓库能按文档安装依赖，并在真实浏览器中独立运行。
- 做了什么：在仓库根目录创建虚拟环境并安装 `requirements.txt`；以 5010 端口启动新服务，检查健康、仪表盘和历史接口；验证桌面与 375px 移动布局、画像表单开关和控制台。
- 观察到的事实：健康接口返回独立服务名，仪表盘为 `application/json`，空历史返回 `days: []`；浏览器无 error/warn，移动端 `scrollWidth` 与视口宽度一致。
- 为什么这样做：单元测试证明函数和接口契约，真实浏览器才能发现 CSP、静态资源、DOM 初始化和响应式布局问题。
- 考虑过的替代方案及取舍：只运行原项目 Python 环境更快，但不能验证 README 安装路径；选择从根目录按文档流程创建虚拟环境。
- 遇到的问题/失败尝试：第一次在 `backend` 目录创建虚拟环境后执行 `pip install -r requirements.txt`，因依赖文件位于仓库根目录而失败；随后按 README 从根目录重建并通过。
- 如何验证：15 条后端测试通过；2 组前端测试和 JavaScript 语法检查通过；真实浏览器控制台为空，375px 下无横向溢出。
- 结论：项目具备独立安装、启动、API 服务和前端运行能力。
- 我应该学会：可复现性需要把文档命令当作产品接口测试；工作目录也是命令契约的一部分。

## 可迁移经验与面试表达

我把个人 AI 助手中的英语学习模块拆成了独立 Agent 项目。原系统的入口、前端和数据目录与多个助手耦合，我先用边界测试定义新服务只能暴露英语 API，再迁移掌握度、错因和间隔复习调度器，重建专用 Flask 入口和前端。系统采用确定性调度器决定学什么、何时复习和题型比例，LLM 负责在约束内生成内容，练习结果再回写长期状态形成闭环。拆分后项目拥有独立配置、测试、文档和 GitHub 仓库，同时未复制个人数据或密钥。

可能追问：

1. 为什么不直接复制原仓库？答：复制会保留其他助手导入、DOM 初始化和数据边界，后续维护仍然耦合；我以 API、配置、数据和测试为四个拆分边界。
2. 为什么规划不用 LLM？答：学习调度需要可解释、可复现和可测试，确定性策略负责决策，LLM 更适合受约束的自然语言生成。
3. 如何保证迁移没有破坏行为？答：先迁移核心测试，再新增独立性契约测试，最后做 API 与真实浏览器验证。
4. 如何避免泄露个人数据和密钥？答：只迁移代码和空数据目录，`.env`、日志和运行时 JSON 全部忽略，提交前扫描暂存区。
5. 后续如何支持多用户？答：引入认证授权和事务数据库，将数据按用户隔离，并把调度任务和 LLM 调用放入受限队列。
